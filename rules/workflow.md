# 開発ワークフロー

IMPORTANT: 以下のフローを必ずこの順番で実行すること。ステップを飛ばしてはならない。

## Step 1: 計画（プランモード）

プランを書く前に以下を確認する:
- 変更対象のコードを Read で読み、現在の実装を把握する
- 変更対象から呼び出している・呼び出されている関連コードを確認する
- 関連する rules/ の仕様を Read で確認する
- 既存の類似実装パターンを Grep で探す
- 不明点があればユーザーに質問する（仮定で進めない）
- UI の現状調査に `RenderPreview` を活用できる（既存 View の `#Preview` をレンダリングして現状を把握する）

上記を踏まえて、プランには以下の設計判断を含めること:

### 設計判断: 配置・構造・順序

変更の規模に関わらず、以下の2点をプランに書く前に検討する。判断結果はプランに明記する（「検討した結果〇〇にした」で十分。該当しない場合は省略してよい）。

- **モジュール配置**: 新しいコードをどのモジュールに置くか。判断基準は「依存の方向に違反しないか」「既存モジュールの責務を逸脱しないか」。新モジュールを切る場合はその理由を書く。モジュール構成の詳細は `rules/architecture.md` を参照する
- **共通化と分離**: 既存コードとまとめるか、分けるか。「片方だけ変更したくなったとき、もう片方に影響なく変更できるか？」で判断する。無理に共通化して分岐だらけになるなら分ける

### リファクタリングガード

設計判断の検討が終わったら、プランを書き始める前に `/refactor-guard` スキルを実行する。

- 調査で特定した変更対象ファイルのパスを引数で渡す
- 判定結果が「⛔ リファクタ先行」の場合、機能実装のプランより先にリファクタリング計画を立てる。リファクタリングと機能実装は別コミットにする
- 判定結果が「✅ 続行可」の場合、そのままプラン作成に進む

このステップは省略してはならない。「できるかどうか」ではなく「やるべきかどうか」を客観的に判定する仕組み。

EnterPlanMode でプランを作成する。

## Step 2: プランレビュー

プランの記述が完了したら、**ExitPlanMode を直接呼んではならない**。
必ず先に `/review-plan-all` スキルを Skill ツールで実行する。
レビュー完了後に ExitPlanMode を呼ぶ。

## Step 3: 実装

プラン承認後、実装に入る前に以下を確認する:
- `rules/` と `references/knowledge.md` を Read で読み、関連する知見がないか確認する
- プランで言及されたファイルを改めて Read で読む（計画時から変わっている可能性がある）
- 型定義・プロトコル・データモデルを実物で確認する（記憶に頼らない）

実装の原則:
- **TDD（Red-Green-Refactor）**: `/tdd` スキルに従って実装する。テストを先に書き、失敗を確認し（RED）、最小の実装で通し（GREEN）、リファクタリングする。テストを一括で書いてから実装に入る「水平スライス」は禁止。1テスト→1実装の垂直スライスで進める
- **SwiftUI**: View 層の実装・変更時は `/swiftui-pro` スキルに従う。モダン API・パフォーマンス・保守性のベストプラクティスを適用する
- **#Preview（必須）**: View 層に変更がある場合、該当 View の `#Preview` マクロを **必ず** `RenderPreview` でレンダリングして視覚確認する。表示に影響しない変更（`.task` の修正等）でも省略しない。新しい View を追加した場合は `#Preview` も追加する。確認観点: レイアウト崩れ（テキスト切れ・要素の重なり・意図しない余白）、状態バリエーションの網羅（空状態・通常・エッジケース）、フレームサイズの適切さ
- **ユーザー動作確認（必須）**: 新しい操作フローや UI 挙動を実装した場合、`build_run_macos` でアプリを起動してユーザーに動作確認を依頼する。計画段階では存在しなかった UI は自動テストだけでは不十分 — 実際に触って確認する

## Step 4: 実装レビュー

実装・テストが完了したら、**コミットしてはならない**。
必ず先に `/review-code-all` スキルを Skill ツールで実行する。

## Step 5: コミット

レビュー完了後、`/commit` スキルでコミットする。

## Codex 指摘の蓄積

`/review-plan-codex` や `/review-code-codex` で MUST / SHOULD の指摘が出たら、`tmp/codex-findings.md` に追記する。

追記形式:
```
## <セッションで何を実装したかの1行要約>（文脈がわかる補足。何のスキル/機能か、どういう設計前提か等）
- [plan|impl] 🔴/🟡 <指摘内容の要約>
  - 判定: ✅ 採用 / ❌ 不要 — <理由を1行で。不要の場合は Codex が何を誤解したかも書く>
  - self で防げたか: Yes/No
  - スコープ: project / common
  - 詳細: <具体的な指摘を1-2行で>
```

ポイント:
- 判定はセッション中に入れる（後から判断するコストが高い）
- 見出しに文脈を入れる（後から見返したとき何の変更か分かるように）

20〜30件溜まったら一括分類し、skill への反映を検討する。
