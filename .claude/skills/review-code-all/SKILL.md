# Implementation Review — Full Chain

実装レビューの全ステップを順次実行し、指摘の反映ループを回す。
**各ステップは前のステップの完了を待ってから実行すること。同時実行は禁止。**
**明示的に「ユーザーに確認」と記載されたステップ以外は、ユーザー確認なしで次のステップへ自動的に進む。**

ユーザーが codex スキップを指示している場合、手順11-12をスキップする。

## 手順

### 0. ビルドとテストを通す

失敗したら修正してから次へ進む。

### 1. UI 動作確認（UI 変更がある場合のみ）

View 層の表示・レイアウト・操作に関わる変更がある場合:

1. **RenderPreview による視覚確認**: 変更した View に `#Preview` があれば `RenderPreview` で順次レンダリングし、`Read` で画像を確認する。確認観点:
   - レイアウト崩れ: テキスト切れ、要素の重なり、意図しない余白
   - 状態の網羅: 空状態 / 通常 / エッジケースの Preview が揃っているか
   - 変更の影響: 今回変更した View 以外の Preview が壊れていないか
   - フレームサイズ: `.frame()` が適切か（巨大/空にならないか）
   問題があれば修正してから次へ進む。`#Preview` が未定義の場合はスキップする（review-code-koecho の制約チェックで検知）。
2. **build_run_macos でアプリ起動**: 操作フロー（ボタン押下、画面遷移等）の確認が必要な場合、`build_run_macos`（または該当プラットフォーム）でアプリを起動し、ユーザーに動作確認を依頼する。RenderPreview のみで十分な場合（表示変更のみ）はスキップしてよい。

自動テストのみでカバーできる変更（ロジック・リファクタ等）の場合は両方スキップする。

### 2. diff サイズを確認する

`git diff --stat` で変更行数を確認する。合計が 1000 行を超える場合、AskUserQuestion で「diff が大きいため Codex レビュー（/review-code-codex）でレート制限に当たる可能性があります。Codex レビューも実行しますか？」と確認する。ユーザーが No なら手順11（/review-code-codex）をスキップして手順13に進む。

### 3. プランファイルの更新（必要な場合のみ）

プランから意図的に変更した箇所がある場合、プランファイルを更新する（該当 Step に変更内容と理由を追記）。

### 4. `/simplify` を Skill tool で実行する

結果を簡潔に報告したら、ユーザー確認を待たずに次の手順へ自動的に進む。

### 5. `/review-code` を Skill tool で実行する

引数（`$ARGUMENTS`）があればそのまま渡す。

### 6. `/review-code-swift` を Skill tool で実行する

引数があればそのまま渡す。

### 7. `/review-code-koecho` を Skill tool で実行する

引数があればそのまま渡す。

### 8. 新規の 🔴 MUST / 🟡 SHOULD 指摘を実装に反映する

- 前回対処済みの指摘の再表現は新規とみなさない
- 判断が必要な指摘は AskUserQuestion でユーザーに確認する

### 9. 新規指摘があった場合 → ループ

- View 層に影響する変更を行った場合 → 手順0に戻る（ビルド・テスト → UI 動作確認 → レビュー）
- View 層に影響しない変更のみの場合 → 手順5に戻る

新規 MUST/SHOULD がゼロになるまでループする。

### 10. 手順2で Codex スキップが決定している場合、手順13に進む

### 11. `/review-code-codex` を Skill tool で実行する

初回はそのまま、2回目以降は `--resume` をつけて呼ぶ。

Codex の出力に 🔴 MUST / 🟡 SHOULD / 🔵 NIT の指摘がある場合（LGTM でない場合）、指摘をメインリポジトリの `tmp/codex-findings.md` に追記する。パスは `"$(dirname "$(git rev-parse --git-common-dir)")/tmp/codex-findings.md"` で解決する（worktree でもメインリポに書く）。ファイルやディレクトリが存在しない場合は作成する。

```markdown
## YYYY-MM-DD impl: <変更の概要（1行）>

- 🔴/🟡/🔵 指摘内容の要約（1行）
- ...
```

### 12. 新規指摘があれば反映し、手順9と同じルールでループ先を判断する

Codex レビューが2回目の場合、自動で反映せず指摘内容をユーザーに提示する。ユーザーがさらにループするか終了するか判断する。
